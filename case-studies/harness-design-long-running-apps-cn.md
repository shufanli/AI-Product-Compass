---

Harness 设计是推动 agentic coding 前沿性能的关键。以下是我们如何在前端设计和长时间自主软件工程中把 Claude 推得更远。

*本文由 Anthropic Labs 团队成员 Prithvi Rajasekaran 撰写。*

过去几个月，我一直在攻克两个相互关联的问题：让 Claude 产出高质量的前端设计，以及让它在无人干预的情况下构建完整应用。这项工作起源于我和同事们早期在前端设计技能和长时间编码 agent harness 上的努力——通过 prompt engineering 和 harness 设计，我们将 Claude 的性能提升到远超基线水平——但两者最终都撞上了天花板。

为了突破，我寻找了在两个截然不同的领域（一个由主观品味定义，另一个由可验证的正确性和可用性定义）都能成立的新型 AI 工程方法。受生成对抗网络（GAN）的启发，我设计了一个包含**生成器**和**评估器** agent 的多 agent 结构。构建一个能可靠评分——且有品味的——评估器，意味着首先要开发一套标准，将"这个设计好吗？"这类主观判断转化为具体的、可评分的条目。

然后我将这些技术应用到长时间自主编码中，从早期 harness 工作中带来了两个经验：将构建任务分解为可处理的块，以及使用结构化产物在会话之间传递上下文。最终成果是一个三 agent 架构——规划器、生成器和评估器——能在数小时的自主编码会话中产出丰富的全栈应用。

## 为什么朴素实现不够好

我们之前已经展示过，harness 设计对长时间 agentic coding 的效果有显著影响。在早期实验中，我们使用一个初始化 agent 将产品规格分解为任务列表，一个编码 agent 逐个功能实现，并在会话之间传递产物以保持上下文。更广泛的开发者社区也收敛到了类似的洞察上，例如"Ralph Wiggum"方法使用 hooks 或脚本让 agent 保持在持续迭代循环中。

但一些问题始终存在。对于更复杂的任务，agent 在长时间运行后仍然容易跑偏。在拆解这个问题时，我们观察到 agent 执行此类任务时的两种常见失败模式。

**第一个是模型在上下文窗口填满时容易失去连贯性**（参见我们关于 context engineering 的文章）。一些模型还表现出"**上下文焦虑**"，即当它们认为自己接近上下文限制时，开始过早地收尾工作。**上下文重置**——完全清空上下文窗口并启动一个新的 agent，结合一个结构化的交接文件来传递前一个 agent 的状态和下一步——能同时解决这两个问题。

这与**压缩（compaction）**不同，压缩是将对话早期部分就地总结，让同一个 agent 在缩短的历史上继续工作。虽然压缩保持了连续性，但它没有给 agent 一个干净的起点，这意味着上下文焦虑仍然可能存在。重置提供了一个干净的起点，代价是交接产物需要包含足够的状态，让下一个 agent 能顺利接手。在早期测试中，我们发现 Claude Sonnet 4.5 表现出的上下文焦虑严重到仅靠压缩不足以支撑良好的长任务性能，因此上下文重置成为 harness 设计的关键。这解决了核心问题，但给每次 harness 运行增加了编排复杂性、token 开销和延迟。

**第二个问题是自我评估**，这是我们之前没有解决的。当被要求评估自己产出的工作时，agent 倾向于自信地赞美自己的工作——即使在人类观察者看来，质量明显平庸。这个问题在设计这类主观任务中尤为突出，因为没有等同于可验证软件测试的二元检查。一个布局是精致还是平庸是主观判断，而 agent 在评价自己的工作时会可靠地偏向正面。

然而，即使在有可验证结果的任务上，agent 有时仍然表现出糟糕的判断力，阻碍其完成任务的表现。**将做工作的 agent 和评判工作的 agent 分开**被证明是解决这个问题的强有力杠杆。分离本身并不会立即消除那种宽容倾向；评估器仍然是一个对 LLM 生成内容倾向于宽容的 LLM。但调优一个独立的评估器使其持怀疑态度，比让生成器批判自己的工作要**容易得多**，而一旦有了外部反馈，生成器就有了具体的改进目标。

## 前端设计：让主观质量可评分

我从前端设计开始实验，因为自我评估问题在这里最明显。在没有任何干预的情况下，Claude 通常会倾向于安全、可预测的布局——技术上功能齐全但视觉上平平无奇。

两个洞察塑造了我为前端设计构建的 harness。首先，虽然审美不能完全归结为一个分数——个人品味总会有差异——但可以通过编码设计原则和偏好的评分标准来提升。"这个设计漂亮吗？"很难一致地回答，但"这个设计是否遵循了我们的良好设计原则？"给了 Claude 具体的评分依据。其次，通过将前端生成和前端评分分离，我们可以创建一个推动生成器产出更强作品的反馈循环。

带着这个思路，我写了四个评分标准，在 prompt 中同时给了生成器和评估器 agent：

- **设计质量（Design quality）：** 设计是否像一个连贯的整体，而不是零件的拼凑？强作品意味着颜色、排版、布局、图像和其他细节结合起来创造出独特的氛围和身份。

- **原创性（Originality）：** 是否有定制决策的证据，还是模板布局、库默认值和 AI 生成的模式？人类设计师应该能识别出深思熟虑的创意选择。未经修改的库存组件——或 AI 生成的典型标志，如白色卡片上的紫色渐变——在这里会失分。

- **工艺（Craft）：** 技术执行：排版层级、间距一致性、色彩和谐、对比度。这是能力检查而非创意检查。大多数合理的实现默认在这里做得不错；失败意味着基本功出了问题。

- **功能性（Functionality）：** 独立于审美的可用性。用户能否理解界面做什么、找到主要操作、在不猜测的情况下完成任务？

我强调了设计质量和原创性，弱化了工艺和功能性。Claude 在工艺和功能性上默认得分就不错，因为所需的技术能力对模型来说比较自然。但在设计和原创性上，Claude 的产出往往平淡无奇。评分标准明确惩罚高度通用的"AI 垃圾"模式，通过给设计和原创性更高权重，推动模型做更多审美上的冒险。

我用 few-shot 示例配合详细的评分分解来校准评估器。这确保了评估器的判断与我的偏好一致，并减少了跨迭代的评分漂移。

我在 Claude Agent SDK 上构建了这个循环，保持编排的简洁性。生成器 agent 首先根据用户提示创建一个 HTML/CSS/JS 前端。我给了评估器 Playwright MCP，让它在评分每个标准和写出详细评语之前直接与活页面交互。实际上，评估器会自行浏览页面，截图并仔细研究实现，然后才产出评估。反馈作为下一次迭代的输入流回生成器。每次生成我运行 5 到 15 次迭代，每次迭代通常将生成器推向更独特的方向，因为它在回应评估器的评语。由于评估器在主动浏览页面而非评分静态截图，每个周期需要真实的时间。完整运行可延伸到四个小时。我还指示生成器在每次评估后做出战略决策：如果分数趋势良好则改进当前方向，如果方法不奏效则转向完全不同的审美。

跨运行来看，评估器的评估在迭代中改善后趋于平稳，仍有提升空间。一些生成是渐进式改进，另一些在迭代之间发生了剧烈的审美转向。

**标准的措辞以我没有完全预料到的方式引导了生成器。** 包含"最好的设计是博物馆级别"这样的措辞，将设计推向了特定的视觉收敛，这表明与标准相关的提示语直接塑造了产出的特征。

虽然分数通常在迭代中提升，但模式并不总是线性的。后期实现总体上更好，但我经常看到自己更喜欢中间某次迭代而不是最后一次。实现复杂度也倾向于在各轮中增加，生成器在回应评估器反馈时尝试更有雄心的解决方案。即使在第一次迭代中，产出也明显好于完全没有提示的基线，这表明标准及其相关语言在任何评估器反馈导致进一步改进之前，就已经将模型从通用默认值引导走了。

在一个值得注意的例子中，我提示模型为一个荷兰艺术博物馆创建网站。到第九次迭代，它产出了一个干净的深色主题虚构博物馆落地页。页面视觉上精致，但基本符合我的预期。然后，在第十个周期，**它完全推翻了这个方案，将网站重新想象为一个空间体验**：一个用 CSS 透视渲染的棋盘格地板的 3D 房间，艺术品以自由形式挂在墙上，通过门道导航在画廊房间之间切换，而不是滚动或点击。这是我之前从未在单次生成中见过的那种创意飞跃。

## 扩展到全栈编码

有了这些发现，我将这种受 GAN 启发的模式应用到全栈开发中。生成器-评估器循环自然映射到软件开发生命周期，其中代码审查和 QA 扮演与设计评估器相同的结构性角色。

### 架构

在我们早期的长时间 harness 中，我们用一个初始化 agent、一个逐功能工作的编码 agent 和会话间的上下文重置，解决了连贯的多会话编码问题。上下文重置是关键突破：harness 使用 Sonnet 4.5，它表现出前面提到的"上下文焦虑"倾向。围绕上下文重置构建良好工作的 harness 是保持模型专注的关键。Opus 4.5 在很大程度上自行消除了这种行为，所以我能够完全从这个 harness 中去掉上下文重置。Agent 在整个构建过程中作为一个连续会话运行，Claude Agent SDK 的自动压缩处理了上下文增长。

在这项工作中，我在原有 harness 基础上构建了一个三 agent 系统，每个 agent 针对我在之前运行中观察到的特定缺口。系统包含以下 agent 角色：

**规划器（Planner）：** 我们之前的长时间 harness 要求用户预先提供详细规格。我想自动化这一步，所以创建了一个规划器 agent，它接收一个简单的 1-4 句提示并将其扩展为完整的产品规格。我提示它在范围上要有雄心，并专注于产品上下文和高层技术设计，而不是详细的技术实现。这个强调是因为担心：如果规划器试图预先指定粒度技术细节而出了错，规格中的错误会级联到下游实现中。让 agent 只约束要交付的产物，让它们在工作中自己找到路径，似乎更聪明。我还要求规划器在产品规格中寻找编入 AI 功能的机会。（见底部附录中的示例。）

**生成器（Generator）：** 早期 harness 的逐功能方法在范围管理上效果很好。我在这里应用了类似的模型，指示生成器按 sprint 工作，从规格中逐个功能推进。每个 sprint 使用 React、Vite、FastAPI 和 SQLite（后来改为 PostgreSQL）技术栈实现应用，生成器在每个 sprint 结束时自我评估工作，然后交给 QA。它还有 git 做版本控制。

**评估器（Evaluator）：** 早期 harness 产出的应用往往看起来很impressive，但当你实际试用时仍有真正的 bug。为了捕获这些，评估器使用 Playwright MCP 像用户一样点击运行中的应用，测试 UI 功能、API 端点和数据库状态。然后它根据发现的 bug 和一套标准（以前端实验为模型，这里改编为覆盖产品深度、功能性、视觉设计和代码质量）对每个 sprint 评分。每个标准有一个硬阈值，任何一个低于阈值，sprint 就失败，生成器会收到关于哪里出了问题的详细反馈。

**在每个 sprint 之前，生成器和评估器协商一个 sprint 合约**：在写任何代码之前就商定那块工作的"完成"标准。这样做是因为产品规格是故意保持高层级的，我需要一个步骤来弥合用户故事和可测试实现之间的差距。生成器提出它要构建什么以及如何验证成功，评估器审查提案以确保生成器在构建正确的东西。双方迭代直到达成一致。

通信通过文件处理：一个 agent 写一个文件，另一个 agent 读取它并在该文件内或用新文件回应，前一个 agent 再去读取。生成器然后根据商定的合约进行构建，然后将工作交给 QA。这让工作忠于规格，同时不会过早过度指定实现。

### 运行 harness

对于这个 harness 的第一版，我使用了 Claude Opus 4.5，将用户提示同时在完整 harness 和单 agent 系统上运行进行对比。我使用 Opus 4.5 因为在我开始这些实验时它是我们最好的编码模型。

我写了以下提示来生成一个复古视频游戏制作器：

> *创建一个 2D 复古游戏制作器，功能包括关卡编辑器、精灵编辑器、实体行为和可玩测试模式。*

下表显示了 harness 类型、运行时长和总成本：

| Harness | 时长 | 成本 |
|---------|------|------|
| 单 agent | 20 分钟 | $9 |
| 完整 harness | 6 小时 | $200 |

harness 贵了 20 倍以上，但产出质量的差异立竿见影。

我期望一个界面，让我可以构建关卡及其组成部分（精灵、实体、瓦片布局），然后点击播放来实际玩关卡。我先打开了单 agent 运行的产出，初始应用看起来符合预期。

但当我点击浏览时，问题开始浮现。布局浪费空间，固定高度面板让大部分视口空着。工作流僵硬。试图填充关卡时提示我先创建精灵和实体，但 UI 中没有任何引导我完成这个序列。更关键的是，**实际游戏是坏的**。我的实体出现在屏幕上但没有任何东西响应输入。深入代码发现实体定义和游戏运行时之间的连接是断的，界面上没有任何指示。

评估完单 agent 运行后，我转向了 harness 运行。这次运行从同一个单句提示开始，但规划器步骤将该提示扩展为一个覆盖十个 sprint 的 16 功能规格。它远远超出了单 agent 运行的尝试范围。除了核心编辑器和播放模式，规格还包括精灵动画系统、行为模板、音效和音乐、AI 辅助精灵生成器和关卡设计器，以及带可分享链接的游戏导出。我让规划器访问了我们的前端设计技能，它读取并用来为应用创建视觉设计语言作为规格的一部分。对于每个 sprint，生成器和评估器协商一个合约，定义该 sprint 的具体实现细节和将用于验证完成的可测试行为。

应用立即显示出比单 agent 运行更多的精致和流畅。画布使用了全视口，面板大小合理，界面有一致的视觉身份，跟踪了规格中的设计方向。我在单 agent 运行中看到的一些笨拙确实仍然存在——工作流仍然没有明确引导你应该先构建精灵和实体再填充关卡，我得自己摸索。这更像是基础模型产品直觉的差距，而不是 harness 设计要解决的问题，不过它确实暗示了 harness 内的定向迭代可以进一步提升产出质量的地方。

在编辑器中工作时，新运行相比单 agent 的优势变得更加明显。精灵编辑器更丰富、功能更完整，有更干净的工具面板、更好的取色器和更好用的缩放控件。

因为我要求规划器将 AI 功能编入规格，应用还内置了一个 Claude 集成，让我可以通过提示来生成游戏的不同部分。这显著加速了工作流。

最大的区别在播放模式。**我实际上能够移动我的实体并玩游戏。** 物理有一些粗糙的边缘——我的角色跳到平台上但与平台重叠，这感觉直觉上不对——但核心功能是有效的，而单 agent 运行没有做到这一点。移动了一会儿后，我确实遇到了一些 AI 游戏关卡构建的局限性。有一面大墙我跳不过去，所以我被卡住了。这表明 harness 可以处理一些常识性改进和边界情况来进一步完善应用。

阅读日志，很明显评估器让实现保持与规格一致。每个 sprint，它遍历 sprint 合约的测试标准，通过 Playwright 操作运行中的应用，对任何偏离预期行为的地方提 bug。合约非常粒度化——仅 Sprint 3 就有 27 个覆盖关卡编辑器的标准——评估器的发现足够具体，可以直接行动而无需额外调查。下表展示了评估器识别的几个问题示例：

| 合约标准 | 评估器发现 |
|---------|-----------|
| 矩形填充工具允许点击拖拽用选中的瓦片填充矩形区域 | **失败** — 工具仅在拖拽起止点放置瓦片，而不是填充区域。`fillRectangle` 函数存在但在 mouseUp 时没有正确触发。 |
| 用户可以选择和删除已放置的实体生成点 | **失败** — `LevelEditor.tsx:892` 处的 Delete 键处理器同时需要 `selection` 和 `selectedEntityId` 被设置，但点击实体只设置了 `selectedEntityId`。条件应该是 `selection \|\| (selectedEntityId && activeLayer === 'entity')`。 |
| 用户可以通过 API 重新排序动画帧 | **失败** — `PUT /frames/reorder` 路由定义在 `/{frame_id}` 路由之后。FastAPI 将 'reorder' 匹配为 frame_id 整数并返回 422: "unable to parse string as an integer." |

让评估器达到这个水平花了不少功夫。开箱即用的 Claude 是一个很差的 QA agent。在早期运行中，我看着它识别出合理的问题，然后说服自己这些不是大问题并批准了工作。它也倾向于表面测试，而不是探索边界情况，所以更微妙的 bug 经常漏掉。调优循环是阅读评估器的日志，找到它的判断与我的偏差，然后更新 QA 的 prompt 来解决这些问题。经过几轮这样的开发循环，评估器的评分才达到我觉得合理的水平。即便如此，harness 产出仍然显示了模型 QA 能力的局限：小的布局问题、某些地方感觉不直观的交互、以及评估器没有深入操作的更深层嵌套功能中的未发现 bug。显然通过进一步调优还有更多验证空间可以捕获。但与单 agent 运行相比——应用的核心功能根本不工作——提升是显而易见的。

### 迭代 harness

第一组 harness 结果令人鼓舞，但也笨重、慢且昂贵。逻辑上的下一步是找到简化 harness 而不降低性能的方法。这部分是常识，部分是一个更一般原则的体现：**harness 中的每个组件都编码了一个关于模型自身无法做什么的假设，这些假设值得压力测试**，既因为它们可能本身就不正确，也因为随着模型改进它们会很快过时。我们的博文《构建有效的 Agent》框定了底层理念为"找到尽可能简单的方案，只在需要时才增加复杂性"，对于任何维护 agent harness 的人来说，这是一个持续出现的模式。

在我第一次尝试简化时，我大幅删减了 harness 并尝试了一些创新的新想法，但我无法复现原版的性能。也变得很难判断 harness 设计的哪些部分实际上是承重的，以及以什么方式承重。基于这个经验，我转向了更有方法的方式，每次移除一个组件并审查它对最终结果的影响。

在我进行这些迭代周期时，我们也发布了 Opus 4.6，这为减少 harness 复杂性提供了更多动力。有充分理由期待 4.6 需要比 4.5 更少的脚手架。来自我们的发布博文："[Opus 4.6] 规划更仔细，更长时间地维持 agentic 任务，能在更大的代码库中更可靠地运行，并且有更好的代码审查和调试技能来捕获自己的错误。"它在长上下文检索上也有显著提升。这些都是 harness 之前被构建来补充的能力。

### 移除 sprint 结构

我首先完全移除了 sprint 结构。sprint 结构曾帮助将工作分解为块，让模型能连贯地工作。鉴于 Opus 4.6 的改进，有充分理由相信模型可以原生处理这项工作，不需要这种分解。

我保留了规划器和评估器，因为两者都持续贡献明显的价值。没有规划器，生成器会欠规划：给它原始提示，它会直接开始构建而不先规划工作，最终创建一个功能不如规划器产出丰富的应用。

移除 sprint 结构后，我将评估器改为在运行结束时做一次性检查，而不是按 sprint 评分。由于模型能力大幅增强，评估器的承重程度因运行而异，其有用性取决于任务相对于模型自身可靠能力的位置。在 4.5 上，那个边界很近：我们的构建处于生成器能独立做好的边缘，评估器在整个构建中捕获了有意义的问题。在 4.6 上，模型的原始能力增强，边界向外移动。过去需要评估器检查才能连贯实现的任务，现在通常在生成器自行处理的范围内，对于这些边界内的任务，评估器变成了不必要的开销。但对于仍处于生成器能力边缘的构建部分，评估器仍然带来了真实的提升。

实际意义是，评估器不是一个固定的"要或不要"决策。当任务超出当前模型可靠的独立能力时，它值得投入成本。

在结构简化的同时，我还添加了提示来改善 harness 将 AI 功能构建到每个应用中的方式，具体是让生成器构建一个能通过工具驱动应用自身功能的真正 agent。这需要真正的迭代，因为相关知识足够新，Claude 的训练数据覆盖较薄。但经过足够的调优，生成器能正确构建 agent。

### 更新后 harness 的结果

为了测试更新后的 harness，我使用以下提示生成了一个数字音频工作站（DAW），一个在浏览器中作曲、录音和混音的音乐制作程序：

> *使用 Web Audio API 在浏览器中构建一个功能完整的 DAW。*

运行仍然耗时且昂贵，约 4 小时，token 成本 $124。

大部分时间花在了构建器上，它在没有 Opus 4.5 需要的 sprint 分解的情况下连贯运行了超过两小时。

| Agent 和阶段 | 时长 | 成本 |
|-------------|------|------|
| 规划器 | 4.7 分钟 | $0.46 |
| 构建（第 1 轮） | 2 小时 7 分钟 | $71.08 |
| QA（第 1 轮） | 8.8 分钟 | $3.24 |
| 构建（第 2 轮） | 1 小时 2 分钟 | $36.89 |
| QA（第 2 轮） | 6.8 分钟 | $3.09 |
| 构建（第 3 轮） | 10.9 分钟 | $5.88 |
| QA（第 3 轮） | 9.6 分钟 | $4.06 |
| **V2 Harness 总计** | **3 小时 50 分钟** | **$124.70** |

与之前的 harness 一样，规划器将单行提示扩展为完整规格。从日志中，我可以看到生成器模型在规划应用和 agent 设计、连接 agent 以及在交给 QA 之前测试方面做得不错。

话虽如此，QA agent 仍然捕获了真实的差距。在第一轮反馈中，它指出：

> 这是一个强大的应用，设计保真度出色，AI agent 扎实，后端良好。主要失败点是功能完整性——虽然应用看起来impressive，AI 集成也运作良好，但几个核心 DAW 功能只是展示性的而没有交互深度：片段不能在时间轴上拖拽/移动，没有乐器 UI 面板（合成器旋钮、鼓垫），没有可视效果编辑器（EQ 曲线、压缩器表）。这些不是边界情况——它们是使 DAW 可用的核心交互，规格明确要求了它们。

在第二轮反馈中，它再次捕获了几个功能差距：

> 剩余差距：
> - 音频录制仍然只是存根（按钮切换但没有麦克风捕获）
> - 未实现通过边缘拖拽调整片段大小和片段分割
> - 效果可视化是数值滑块，不是图形化的（没有 EQ 曲线）

生成器在独自工作时仍然容易遗漏细节或保留功能存根，QA 在捕获这些最后一公里问题上仍然有价值，让生成器去修复。

基于提示，我期望一个程序能让我创建旋律、和声和鼓点模式，将它们编排成歌曲，并在过程中得到集成 agent 的帮助。最终应用具备了功能性音乐制作程序的所有核心部件：在浏览器中运行的编排视图、混音器和传输控制。除此之外，我能够完全通过提示组合一个短歌曲片段：agent 设置了节拍和调性，铺下旋律，构建鼓轨，调整混音器电平，并添加了混响。歌曲作曲的核心基元都在，agent 能自主驱动它们，使用工具从头到尾创建一个简单的制作。你可能会说它还没有完美到位——但已经在接近了。

## 下一步是什么

随着模型持续改进，我们大致可以期待它们能够工作更长时间、处理更复杂的任务。在某些情况下，这意味着围绕模型的脚手架随时间变得不那么重要，开发者可以等待下一个模型，看某些问题自行解决。另一方面，模型越好，就有越多空间开发 harness 来实现超出模型基线能力的复杂任务。

带着这个思路，这项工作有几个值得向前带的经验。**试验你正在构建的模型、在真实问题上阅读它的 trace、调优它的性能以达到你期望的结果**，始终是好的实践。在处理更复杂的任务时，有时可以通过分解任务并对问题的每个方面应用专门的 agent 来获得提升空间。当新模型到来时，重新审视 harness 通常是好的实践，剥离不再对性能承重的部分，添加新部分以实现之前不可能的更大能力。

从这项工作中，我的信念是：**有趣的 harness 组合空间不会随着模型改进而缩小。相反，它会移动，而 AI 工程师的有趣工作是不断找到下一个新颖的组合。**
