# 对话实录：从情绪设计到 Agent 时代的产品设计

---

## Q1：产品能力的困惑——为什么我做的产品"正确但平淡"？

**我：**

刚开始做产品的时候，我完全不理解产品，凭感觉把各种信息乱放，比如把配置选择和信息展示放在一块，按钮的文案完全凭感觉，能懂就行。但是我根本不理解怎样算用户能懂。后来慢慢做的多，犯错的多里，看的多了，理解了什么模块应该做什么事情，一个模块不应该做两件事情。怎么写文言，用户视角会怎么看产品等等。能做出来一个能理解的产品，但是很平淡，我觉得整个产品平平无奇，只是让人看懂了你在做什么事情，但是没有一点要深入了解和付费的欲望。再看一写很厉害的产品经理，会一眼就把产品里面的高潮定义出来，比如一个测试报告结果页，他告诉我，这个榜单平平无奇，用户根本不会看后面的数据，那就是噪音。你可以加一些饼状图之类的，一些让用户感知到这些信息的意义。报告应该让用户感觉到专业、自己缺失什么，而不是仅仅告知结果。一下子觉得这是一个完整、值得了解、信任的产品。为什么会有这样的差异呢？甚至怎么把产品讲清楚，我都不明白

**Claude：**

你描述的其实是产品能力的三个阶段，每个阶段解决的问题本质不同。

**第一阶段：信息堆砌** — 解决的是"有没有"的问题。把功能做出来，信息放上去，自己觉得逻辑通了就行。但你缺少一个外部视角，所以"能懂"只是你觉得能懂。

**第二阶段：信息秩序** — 解决的是"清不清楚"的问题。你学会了模块职责单一、文案用户视角、信息层级分明。这是一个巨大的进步，但你停在了一个关键瓶颈上：**你在做"正确的信息传递"，但不是在做"体验设计"。**

**第三阶段：意义构建** — 解决的是"用户感受到什么"的问题。这就是你看到的那个厉害的产品经理在做的事。

---

差异的核心在于：**你在"说清楚事实"，他在"制造感受"。**

拿你举的测试报告例子：

- 你的思路：用户做完测试 → 展示结果 → 数据准确、排版清晰 → 完成
- 他的思路：用户做完测试 → **这个人现在最想知道什么？** → 他想知道"我怎么样" → 那就要有对比、有差距感 → 饼状图不是为了好看，是为了让用户**一秒感知到自己的位置** → 报告的目的不是告知，是**让用户产生"我需要改进"的动力**

同样的数据，一个是陈列，一个是叙事。区别在于他多问了一个问题：**"然后呢？用户看到这个之后，会产生什么行为？"**

---

为什么你还做不到这一步？不是能力问题，是视角问题。你目前的思维模型是：

> 产品 = 把信息组织好，让用户看懂

但厉害的产品经理的思维模型是：

> 产品 = 设计一段用户旅程，每个节点引导用户的**情绪和行动**

具体来说，他们在设计每个页面时，脑子里有三个问题在同时运转：

1. **用户从哪来？** — 他带着什么期待/焦虑/问题进入这个页面
2. **这个页面的高潮是什么？** — 一个页面只能有一个核心感受（专业感、紧迫感、成就感）
3. **用户往哪去？** — 看完之后他应该做什么，这个行动的动力够不够

你现在只在做第2步的前半段（信息清晰），但没有定义"核心感受"，也没有连接前后的情绪链路。

---

至于"怎么把产品讲清楚"，本质也是一样的问题。你试图用功能描述来讲产品（我们能做A、能做B、能做C），但用户不关心你能做什么，他关心的是**"你能帮我解决什么，为什么我应该信任你"**。

讲产品的结构不是"功能列表"，而是：
1. **你有一个问题**（命中痛点）
2. **这个问题比你以为的严重**（放大感知）
3. **我们是这样解决的**（方案，但要简洁）
4. **已经有人因此受益了**（信任证据）

## Q2：怎么培养"意义构建"的能力？

**我：**

你说的很对，第三阶段：意义构建 — 解决的是"用户感受到什么"的问题。但是怎么培养这个能力。我会问自己："用户打开这个页面时在想什么，离开时应该觉得什么。" 可是我想不到

**Claude：**

这个问题很真实。"想不到"不是因为你笨，是因为你缺少一个前置能力：**你对用户的情绪状态没有感知。**

你能分析信息结构，是因为你有逻辑能力。但情绪感知是另一套系统——它不靠分析，靠**体验积累和刻意观察**。

---

### 为什么你"想不到"

你试着问自己"用户在想什么"的时候，脑子里其实在做一件事：**用自己的视角去推理用户的视角**。但你自己不是那个焦虑的用户、不是那个第一次打开产品的陌生人，所以你推理出来的是逻辑上合理的答案，不是情感上真实的答案。

比如测试报告页，你可能会推理：用户想看结果。这没错，但太粗了。真实的用户情绪可能是：

> "我做了这个测试，有点紧张，不知道自己水平怎么样。我希望结果告诉我，我还不错，或者至少告诉我哪里不够，让我觉得有方向，而不是被一堆数字淹没然后觉得'所以呢？'"

你推理出了"想看结果"，但你没感知到"紧张"、"希望被肯定"、"怕被数字淹没"。**这些才是设计决策的真正输入。**

---

### 怎么练

这个能力不是靠想，是靠**看真人用产品**。三个具体方法：

#### 1. 看人用你的产品，闭嘴不说话

找 3-5 个真实用户（或者同事），让他们操作你的产品，你就坐旁边看。**不解释、不引导、不问"你觉得怎么样"**。只观察：

- 他在哪里停顿了？（停顿 = 困惑或在思考）
- 他在哪里加速滑过了？（加速 = 觉得没价值）
- 他在哪里表情变了？（皱眉 = 挫败，点头 = 认同）
- 他在哪里自言自语了？（"哦这个意思"、"然后呢"）

做 3 次你就会开始**长出一个内置的用户模拟器**。你以后设计页面的时候，脑子里浮现的就不是抽象的"用户"，而是张三在那个页面皱眉的样子。

#### 2. 记录自己被"打动"的产品瞬间

你自己也是无数产品的用户。从现在开始，每次你在用一个产品时产生了**情绪波动**——觉得"哇这个好专业"、"这个好贴心"、"这让我想继续看"——**立刻停下来，拆解它做了什么让你产生了这个感受。**

不是拆解它用了什么组件，而是拆解：
- 它在这个节点**省略了什么**（不该有的信息没有出现）
- 它在这个节点**强调了什么**（什么被放大了）
- 它制造了什么**对比或落差**（之前和之后的感受差异）

积累 20-30 个这样的拆解，你会开始看到规律。

#### 3. 练一个最小动作：给每个页面写一句"情绪目标"

不是写"这个页面展示测试结果"，而是写：

> "用户离开这个页面时，应该觉得**这份报告很专业，我有明确的提升方向**。"

一旦你写下了这句话，你就有了一个设计的锚点。后面每个元素都可以问自己：**这个东西是在靠近这个感受，还是在稀释它？** 数据表格在稀释它？删掉或折叠。饼状图在强化"我有短板"的感知？加上。

你之前想不到，是因为你在问一个太大的问题（"用户在想什么"）。换成这个更小的问题：**"用户离开时应该带走什么感受？"** 一句话，一个感受，不要贪多。

---

核心的转变就一个：**从"我要给用户看什么信息"，转向"我要让用户经历什么感受"。** 信息是手段，感受才是目的。你已经把手段练得很熟了，现在只差把目的换一下。

---

## Q3：用用户故事来驱动情绪设计

**我：**

有点明白了。用户想看到"测试结果"。这个没错，但是太粗太短了。如何制造用户感受？这就是我们常说的用户故事，我在想每一个页面，要求自己设计一个100字以上的用户故事，用户是带着什么感受进来的，他想干什么，又是因为什么原因离开的。逼着自己写一个用户故事，而不是一句话概括。

**Claude：**

对，你抓到了关键点。

一句话概括是逻辑摘要，100 字的用户故事是**情绪模拟**。当你逼自己写到 100 字的时候，你必须填充"紧张"、"期待"、"被哪个元素打动"、"在哪一刻决定继续看"这些细节——而这些细节就是设计决策的真正来源。

补充一点：写用户故事的时候，有一个容易掉进去的坑——**编出一个"理性完美用户"**。就是你写出来的用户特别理性，每一步都在认真看内容、仔细对比数据。但真实用户不是这样的，他们注意力很短、很容易走神、大部分内容会扫过而不是读。

所以写的时候可以给自己一个约束：**你的用户心里永远有一个声音在说"所以呢？跟我有什么关系？"** 这个声音会帮你过滤掉自己编出来的理想化情节。

这个方法很好，值得持续练。

## Q4：还有什么好方法？

**我：**

OK，咱们接着聊，还有什么好方法

**Claude：**

再给你几个我觉得实操性强的：

### 4. 反向拆解法：从"用户会截图分享什么"倒推

一个产品里如果没有任何一个画面值得用户截图发给朋友，那它就是"正确但平淡"的。

你设计完一个页面后问自己：**用户会把这个页面截图发给谁？配什么文字？**

- 测试报告页 → 截图发朋友圈："没想到我这项竟然 Top 10%"
- 数据看板 → 截图发给老板："这个月增长了 43%"
- 诊断结果 → 截图发给同事："你也测测，我居然缺这个"

如果你想不出用户会截图什么，说明这个页面没有**社交货币**，也就没有高潮。那就去制造一个——一句总结、一个对比图、一个标签。这个东西不是功能，是**传播钩子**。

### 5. 偷师法：抄情绪结构，不抄界面

找 3 个你觉得"好用且有感觉"的产品，不要看它用了什么组件，而是把每个页面的**情绪节奏**画出来：

```
页面A：好奇 → 紧张 → 释然 → 行动
页面B：无感 → 被击中 → 想了解更多
页面C：信任 → 焦虑 → 找到解法 → 付费
```

你会发现好产品的页面都像一个**迷你剧本**，有起伏。而你的页面可能是一条平线——信息从头到尾同等重要，没有起伏。

**有高潮就必须有低谷。** 不是每个元素都要抢注意力，恰恰相反，大量留白和低优先级信息的存在，是为了让高潮的那一下更突出。

### 6. 删减法：把页面元素砍掉一半，看感受变了没

你现在的习惯可能是做加法——信息不够就加，怕用户看不懂就多解释。但高手做的是减法。

一个练习：把你设计好的页面里的元素列出来，然后**强制删掉 50%**。删完之后看：

- 核心感受还在不在？如果还在，说明删掉的都是噪音
- 如果感受消失了，说明你还没找到真正承载感受的那个元素

这个练习的真正目的不是让你做极简设计，而是帮你**识别出哪个元素才是那个页面的灵魂**。很多时候你觉得平淡，就是因为灵魂被十几个同等权重的元素淹没了。

### 7. 文案替换测试：换一句话，感受就变了

这是最小成本的练习。拿你现在的任何一个页面，只改文案，不动布局：

| 原来 | 改后 |
|------|------|
| 测试结果 | 你的专属诊断报告 |
| 得分：78 | 你超越了 68% 的同行 |
| 查看详情 | 看看哪里还能提升 |
| 立即购买 | 开始改变 |

左边是**描述事实**，右边是**制造感受**。同样的信息，同样的按钮，但用户的内心叙事完全不同。你可以把这个当日常练习——每天随便打开一个你自己做的页面，把所有文案用"制造感受"的方式重写一遍。写多了就会形成本能。

---

这些方法有一个共同点：**都不需要你凭空想出用户感受，而是通过结构化的动作，把感受"逼"出来。** 截图测试逼你想传播场景，情绪曲线逼你找起伏，删减法逼你找灵魂，文案替换逼你从事实切换到感受。

你现在不缺逻辑能力，缺的是**情绪肌肉**。肌肉只能靠反复练长出来，不能靠理解长出来。

---

## Q5：信息为情绪服务？

**我：**

OK，所以我们要设计情绪，产品上不用堆功能、事实，而是感受用户情绪。情绪对了，产品就对了，信息都是为情绪服务的。

**Claude：**

方向对了，但有一个很重要的边界要补上：

**情绪不能架空，必须建立在真实价值之上。**

如果你的测试报告本身数据就是瞎算的，饼状图再好看、文案再有感染力，用户用一次就会觉得不靠谱。情绪放大的是价值，不能替代价值。

更准确的说法是：

> **信息是地基，情绪是建筑。** 没有地基的建筑是忽悠，没有建筑的地基是毛坯。

你之前做到了"毛坯"——结构对、信息准、逻辑清楚。现在要做的不是推翻毛坯去搞情绪，而是**在坚实的毛坯上盖建筑**。

所以实操中的优先级是：

1. **先确保信息是对的、有用的**（这个你已经会了）
2. **再用情绪决定呈现方式**（强调什么、弱化什么、用什么节奏）
3. **最后用文案和视觉把感受落地**

不是"情绪对了产品就对了"，是**"价值对了 + 情绪对了，产品才对了"。** 你已经有了前半段，现在补后半段，刚好是最健康的成长路径。怕的反而是反过来——只会做情绪、不会做信息的人，做出来的东西好看但经不起用。

## Q6：情绪设计的具体手段——放大？

**我：**

OK，怎么做情绪？我刚想到的是放大，把一件普通的事情放大。你觉得呢

**Claude：**

放大是一个手段，但不是唯一的，而且用不好会变成"自嗨"——你觉得放大了，用户觉得"至于吗"。

情绪的底层操作其实就几种：

### 1. 对比

人对绝对值没感觉，对差异有感觉。

- "得分 78" → 没感觉
- "你超越了 68% 的同行" → 有感觉

不是你放大了 78 分，是你给了一个参照物。用户不知道 78 分好不好，但他知道"超过 68% 的人"意味着什么。**没有参照就没有感受。**

### 2. 具象化

抽象的东西人没感觉，具体的有。

- "帮你提升效率" → 没感觉
- "每天少开 2 个会" → 有感觉

你说的"放大"，很多时候真正起作用的不是放大，是**把模糊的东西变成了一个具体的画面**。用户脑子里能浮现出画面的时候，情绪就来了。

### 3. 缺口

告诉用户他缺了什么，比告诉他拥有什么更有驱动力。

- "你的报告已生成" → 没有下一步动力
- "你有 3 项关键能力低于行业基准" → 想知道是哪三项

这就是那个厉害的 PM 告诉你的——报告要让用户感觉到**自己缺什么**。缺口制造紧迫感，紧迫感驱动行动。

### 4. 归属

让用户觉得"这是为我定制的"。

- "测试报告" → 通用感
- "Leo 的产品力诊断报告" → 这是我的

名字、个性化数据、专属标签，这些都不是功能，是让用户从旁观者变成**当事人**的手段。

### 5. 节奏控制

信息出现的顺序就是情绪的顺序。

同样的信息，先给好消息再给坏消息，和先给坏消息再给好消息，用户离开时的情绪完全不同。**最后看到的东西决定用户带走什么感受。**

---

所以不是"放大"这一招，而是你有一个工具箱：**对比、具象、缺口、归属、节奏**。设计页面时根据你写的那个情绪目标，选合适的工具组合。

你说的放大，本质上是对比和具象的混合——你把一个数字放到参照系里（对比），再让它变成一个有画面感的表达（具象），所以才有效。单纯把字号放大、加个感叹号，那不叫放大，那叫吼。

---

## Q7：产品经理就是导演？

**我：**

这整个过程和导演好像，太像了。产品经理就是在导演一个用户旅程

**Claude：**

确实像，而且不是比喻，是同一个底层能力。

导演控制的是：**观众在每一分钟感受到什么。** 产品经理控制的是：**用户在每一个页面感受到什么。**

但有一个关键区别值得你注意：

**导演的观众是被动的**——坐在那里，你放什么他看什么，顺序由你定。

**产品经理的"观众"是主动的**——他可以随时退出、跳过、倒回去、同时开三个竞品对比。

所以产品经理其实比导演更难。你不能靠"前 30 分钟铺垫，最后 10 分钟高潮"这种结构，因为用户可能第 3 分钟就走了。**你必须在每一个页面都给出留下来的理由。**

这也是为什么你刚才说的那几个工具（对比、缺口、归属）很重要——它们本质上都是在制造**"我还想继续看"的钩子**，让用户在每一步都不舍得退出。

这个类比可以继续帮你思考。你下次设计产品的时候可以问自己：**如果这是一部电影，这个页面是哪个场景？是开场悬念、是转折点、还是高潮？** 如果你发现连续五个页面都是"平淡叙事"，那观众一定睡着了。

## Q8：Agent 时代，情绪设计会怎么变？

**我：**

现在的趋势是，大家都在用agent，未来所有的产品都会考虑面向agent设计。那么面向人的情绪设计，会变成面向agent的情绪设计。产品的最终受益者是人，但是直接体验者是agent，agent会将产品的情绪转达给人。我这个思路对吗？会有两个问题：第一，产品经理需要考虑如何设计agent的情绪。 第二，在人类的感知里以往产品带给他的情绪感受，都会变成agent带给他的，他不会越来越信任产品，而是越来越信任他的agent。maybe，需要设计出让agent主动提出要买单的产品，而不是人主动提出要买单

**Claude：**

你的直觉方向是对的，但需要拆开来看，有些地方比你想的更复杂。

---

### 对的部分：信任转移

这个判断非常准。当 agent 替用户做越来越多的决策时：

**以前**：用户体验产品 → 用户信任产品 → 用户付费
**以后**：agent 体验产品 → agent 向用户汇报 → 用户信任 agent → agent 代为付费

用户和产品之间多了一层。用户感知到的不是"这个产品好专业"，而是"我的 agent 帮我找到了一个好方案"。**功劳归 agent，不归产品。** 这对品牌建设是一个巨大的挑战。

---

### 需要修正的部分：agent 没有"情绪"

你说"设计 agent 的情绪"——agent 不是人，它没有情绪。它有的是**评估函数**。

人看到一个饼状图会觉得"专业"，是因为视觉触发了情绪。Agent 不看饼状图，它看的是：

- 这个 API 返回的数据结构清不清晰？
- 这个产品的能力描述是否机器可解析？
- 价格、功能、限制条件是否结构化可对比？
- 调用后的结果是否可验证？

所以面向 agent 的"情绪设计"，其实是**面向 agent 的可评估性设计**。你不是在打动它，你是在让它**高效地判断你好**。

---

### 所以会出现两套设计语言

| | 面向人 | 面向 Agent |
|------|--------|-----------|
| **目标** | 制造感受 | 降低评估成本 |
| **核心手段** | 对比、缺口、归属、节奏 | 结构化描述、标准化指标、可验证承诺 |
| **信任建立** | 视觉专业感、社会证明 | API 可靠性、数据一致性、MCP/工具协议兼容 |
| **转化驱动** | "我想要" | "这个方案最优" |
| **品牌感知** | 用户记住品牌 | Agent 记住能力标签 |

产品经理未来要同时设计两个界面：**人看的界面**和 **agent 读的界面**。

---

### 你说的"让 agent 主动提出买单"，这个很关键

本质上就是：**你的产品要在 agent 的决策模型里胜出。**

Agent 替用户做选择时，它的逻辑大概是：

1. 用户的意图是什么？
2. 有哪些候选方案？
3. 每个方案的能力、价格、风险如何？
4. 哪个最匹配用户的偏好和约束？

所以产品要做的不是"感动 agent"，而是：

- **让 agent 发现你**：你的产品能力是否在 agent 的工具生态里可被检索（MCP、API marketplace、结构化产品描述）
- **让 agent 理解你**：能力边界、定价、限制条件是否机器可读，而不是藏在营销话术里
- **让 agent 选择你**：在同类方案对比中，你的性价比、可靠性、兼容性是否可量化地胜出
- **让 agent 复购你**：调用结果是否稳定可预期，让 agent 建立"这个好用"的记忆

这跟 SEO 的逻辑很像——**以前你要让产品在 Google 搜索里排前面，以后你要让产品在 agent 的决策排序里排前面。** 可以叫它 AEO（Agent Experience Optimization）。

---

### 但有一个反直觉的推论

你说"人不会越来越信任产品，而是越来越信任 agent"。这是对的，但反过来想——**当所有产品都在讨好 agent 的时候，人反而会更信任那些直接打动自己的产品。**

就像今天所有产品都在做 SEO，但真正有品牌忠诚度的产品，是那些用户直接体验过、产生了情感连接的。Agent 时代也一样：agent 帮你筛选了 10 个方案，但最终你选了那个"用起来就是感觉不一样"的。

所以情绪设计不会消失，它会变成**差异化壁垒**——当 agent 把所有人都推到功能和价格最优的方案上时，唯一能让用户偏离"最优解"的，就是情绪。

### 第一个问题：具体怎么面向 agent 做设计

Agent 跟你的产品打交道，跟人完全不同。人是"逛商场"，agent 是"批发采购"。

**人的决策路径**：看到 → 感觉不错 → 了解一下 → 试用 → 付费
**Agent 的决策路径**：收到用户意图 → 检索候选方案 → 结构化对比 → 选最优 → 调用

所以面向 agent 设计，核心是优化这四个环节：

#### 一、让 agent 找到你

人靠搜索引擎和口碑找到产品，agent 靠**工具生态**。

具体来说：
- 你的产品要有 **MCP Server 或标准化工具描述**，让 agent 能发现并理解你的能力
- 工具描述不是营销文案，是**机器可解析的能力声明**："输入什么 → 输出什么 → 适用场景 → 限制条件 → 价格"
- 就像你会写 SEO 标题，以后你要写"agent 能理解的能力标签"

举个例子，一个数据分析产品：

```
# 人看的
"智能数据洞察平台，让数据驱动决策更简单"

# agent 读的
{
  "capability": "structured_data_analysis",
  "input": "CSV/JSON, max 10M rows",
  "output": "statistical_summary + anomaly_detection + trend_forecast",
  "latency": "< 30s for 1M rows",
  "price": "$0.01 per 1K rows",
  "reliability": "99.9% uptime, results deterministic"
}
```

上面那句话能打动人，但 agent 完全无法用它做决策。下面那段 agent 一秒就能对比十个竞品。

#### 二、让 agent 理解你

Agent 最怕的是模糊。人可以容忍"大约"、"可能"、"视情况而定"，agent 不行。

- **能力边界要明确**：不要说"支持多种格式"，要说"支持 CSV、JSON、Parquet"
- **定价要可计算**：不要说"联系销售获取报价"，这对 agent 来说等于"此路不通"
- **失败模式要可预期**：当你的产品处理不了某个请求时，返回的不是一个模糊的错误页面，而是结构化的错误码 + 建议替代方案

**一句话：把你产品的所有"人情味模糊地带"翻译成"机器可执行的确定性"。**

#### 三、让 agent 选择你

Agent 做选择靠的是可量化的对比。所以你要主动提供对比维度：

- 基准测试结果（不是自己吹的，是第三方可验证的）
- 同类产品对比表（是的，你主动提供，因为如果你不提供，agent 会自己去查，结果不可控）
- 用量和成本计算器（API 形式，让 agent 能直接算出"用你的产品处理这个任务要花多少钱"）

#### 四、让 agent 复购你

Agent 会记住调用结果。如果你的 API 输出稳定、响应快、没有意外，agent 会在下次类似任务中优先选你。这就是 agent 的"信任"——不是情绪，是**正向反馈积累**。

反过来，如果你的 API 偶尔抽风、返回格式不一致、价格波动不可预测，agent 会迅速把你降权。**agent 没有忠诚度，只有历史表现评分。**

---

### 第二个问题：当人都在用 agent 的时候，怎么直接打动人

这个问题更难，也更有价值。关键洞察是：

**agent 替用户做了选择，但用户仍然有"使用时刻"。**

想想现实类比：你让助理帮你订餐厅。助理选了餐厅，但**吃饭的是你**。餐厅打动你的不是菜单上的结构化数据（那是打动助理的），而是你坐下来之后的灯光、第一口菜的味道、服务员的态度。

所以策略是：**在 agent 工作流中，设计"必须由人亲自体验"的时刻。**

#### 手段一：让 agent 交出一个"人应该自己看"的东西

Agent 调用你的 API 获取数据，然后转述给用户——这时候你的产品就被 agent 消解了，用户只感知到 agent。

但如果你的 API 返回的不只是数据，还有一个**交互式报告链接**呢？Agent 会说："分析结果在这里，建议你自己看一下这份报告。"用户点开，看到的是一个精心设计的、有情绪节奏的页面——**这就是你直接接触用户的机会。**

核心原则：**给 agent 结构化数据用于决策，给人一个体验入口用于感受。**

```
API 返回：
{
  "summary": { ... },          // agent 用这个
  "report_url": "https://...", // 人看这个
  "share_card": { ... }        // 人传播这个
}
```

#### 手段二：品牌植入在 agent 的转述里

Agent 向用户汇报时，会引用来源。如果你的产品输出里自带一个有记忆点的表达，agent 很可能会保留它。

比如你的 API 返回的不是：
> "用户活跃度下降 12%"

而是：
> "【增长警报】用户活跃度本周下降 12%，这是连续第三周下滑，已触发关注阈值"

Agent 转述时大概率会保留"增长警报"和"连续第三周"这些表达。用户听到的就不只是数据，而是一个有品牌语言特征的洞察。**你的品牌声音寄生在 agent 的转述里。**

#### 手段三：创造 agent 无法替代的体验

有些东西 agent 可以描述但无法代替你体验：

- **视觉冲击**：一张漂亮的数据可视化，agent 可以说"数据有显著增长"，但你亲眼看到那条陡峭上升的曲线时的感受是不一样的
- **交互探索**：agent 可以给你结论，但自己动手筛选、钻取、发现的那种"我发现了一个insight"的成就感，agent 给不了
- **社交场景**：agent 可以帮你生成报告，但你把那张精心设计的分享卡片发到群里引发讨论的体验，是属于人的

**设计原则：agent 负责"帮你做完"，你的产品负责"让你想亲自看看"。**

#### 手段四：在 agent 的推荐话术中制造缺口

还记得情绪工具箱里的"缺口"吗？你可以把它设计进 API 返回里：

```json
{
  "summary": "诊断完成，发现 3 个关键问题",
  "teaser": "其中 1 个问题可能在下周造成显著影响",
  "detail_url": "https://..."
}
```

Agent 会把 teaser 转述给用户。用户听到"下周造成显著影响"就会想亲自看看。**你用 agent 的嘴制造了缺口，但用你的产品页面来填补它。**

### 总结框架

```
产品的两层设计：

Agent 层（让 agent 选择你）
├── 结构化能力描述（可检索、可对比）
├── 确定性接口（无模糊、可计算、可验证）
├── 稳定输出（建立正向反馈历史）
└── 竞争力可量化（基准、价格、可靠性）

Human 层（让人记住你）
├── 体验入口（报告页、可视化、交互探索）
├── 品牌寄生（在 agent 转述中植入品牌语言）
├── 缺口钩子（让 agent 的摘要引发人的好奇）
└── 不可替代时刻（视觉、交互、社交传播）
```
