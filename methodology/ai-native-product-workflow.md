# AI Native 产品工作流：拥抱不确定性

AI 时代的产品工作流正在从"精确定义后执行"转向"框架先行、快速迭代"，产品经理必须学会在不确定性中推进。

## 范式转变：从确定性到不确定性

### 传统产品工作流

- PRD 精确定义每一个交互细节、每一个角标
- 数据流清晰：知道每个数据的来源和去向
- 开发按照规格说明确定性地实现
- 流程是线性的：需求确认 → 设计 → 开发 → 测试 → 上线

### AI Native 产品工作流

- 人定框架，AI 批量生成 PRD，Agent 之间互相 Review 直至逻辑自洽
- 不等 PRD 完全 ready 就启动开发，边跑边改
- 修改成本极低，迭代速度极快
- 流程是并行的：PRD 生成 / 开发 / Review 同时推进

## 三大核心不确定性

### 1. 大模型输出不确定

**问题：** 大模型的输出具有随机性，同样的 Prompt 可能产生不同结果，且原始输出往往不能直接交付给用户。

**应对策略：**

- **输出封装层：** 在模型原始输出和用户之间建立封装层，包括格式化、校验、兜底逻辑（Fallback）
- **Guardrails 机制：** 通过 Output Parser、内容过滤、格式约束等手段收窄输出空间
- **多次采样 + 筛选：** 对关键输出采用多次生成、自动评估、择优展示的策略
- **用户预期管理：** 在产品设计层面让用户理解 AI 输出的概率性本质，例如提供"重新生成"按钮、置信度标识

### 2. 可实现性不确定

**问题：** 一个功能开发能不能做、大模型能不能做、做出来效果好不好——这些在传统产品中可以通过数据流分析确认，但在 AI 产品中往往需要实验才能知道。

**应对策略：**

- **Spike 验证前置：** 对不确定的 AI 能力，先用最小 Prototype 验证可行性（Prompt 能不能跑通、效果是否达标），再写完整 PRD
- **分级承诺：** 将功能拆分为"确定可实现"和"需要验证"两类，给后者设定明确的验收标准（Eval Metrics）和止损点
- **Eval 驱动开发：** 建立自动化评测体系，用量化指标（准确率、延迟、成本）替代主观判断，让"好不好"有客观标准
- **降级方案前置：** 每个 AI 能力点都预设降级策略——模型不可用时的兜底方案是什么

### 3. PRD 内容不确定（审查者盲区）

**问题：** AI 生成的 PRD 可能包含幻觉，而人作为审查者（而非撰写者）存在认知偏差——容易审查"这么做对不对"，却忽略"该不该做"和"还能做什么"。

**应对策略：**

- **结构化审查清单：** 强制自己在审查时回答三个问题：
  1. **对不对** — 现有逻辑是否自洽、是否有幻觉
  2. **该不该** — 这个需求的优先级和 ROI 是否合理
  3. **全不全** — 是否遗漏了边界场景、异常处理、关联需求
- **Agent 交叉审查：** 让多个 Agent 从不同角色（用户、开发、测试、业务）审查同一份 PRD，弥补单一视角的盲区
- **Checklist as Code：** 将审查规则固化为 Prompt 模板或自动化检查脚本，降低人工审查的遗漏率
- **溯源标注：** 要求 AI 在 PRD 中标注关键结论的依据来源，无来源的结论重点审查

## 应对不确定性的三个能力

### 1. 深入理解大模型与 AI 架构

不是当黑盒用，而是要理解背后的机制：

- **模型层：** 了解 Token 机制、上下文窗口限制、Temperature 对输出的影响、模型的能力边界
- **架构层：** 理解主流 AI 应用架构（如 RAG、Agent、Multi-Agent），知道 Session 管理、Memory 机制、Tool Use 的实现方式
- **工程层：** 了解 Prompt Engineering 最佳实践、向量检索原理、模型部署与调用链路

> 例如：理解 OpenClaw 等架构中 Session 的实现方式，才能准确评估"多轮对话"功能的技术约束和产品边界。

### 2. 学会和 Agent 对齐，而不只是和人对齐

传统 PM 的核心能力是和人沟通对齐——和设计师、开发、业务方。AI 时代新增了一个关键对齐对象：Agent。

- **Prompt 即需求文档：** 给 Agent 的指令就是一种 PRD，需要清晰、无歧义、结构化
- **Context 管理：** 理解 Agent 的上下文限制，学会拆分任务、控制信息密度
- **反馈循环：** 像调试代码一样调试 Agent 的输出——分析哪里出了问题、调整 Prompt、重新运行
- **多 Agent 协作编排：** 学会设计 Agent 之间的协作流程，定义各 Agent 的角色、输入输出和交互规则

### 3. 适应"先出结果再迭代"的节奏

传统流程：PRD Ready → 评审通过 → 排期开发 → 测试上线
AI Native 流程：框架确定 → AI 生成 PRD → 快速检查 → 交给开发 Agent → 持续修正 → 持续同步

**关键转变：**

- **修改成本极低：** AI 生成和修改 PRD 的速度远超人工，不必追求一次到位
- **并行推进：** PRD 完善和开发可以同步进行，发现问题即时修正、即时同步
- **CI Spec 成为核心：** 当迭代速度极快时，CI（持续集成）的规格定义变得至关重要——它是确保快速迭代不失控的护栏

## AI Native 产品工作流实践

### 推荐工作流

```
┌─────────────┐
│  人：定义框架  │  ← 产品愿景、核心功能、约束条件
└──────┬──────┘
       ▼
┌─────────────┐
│ AI：批量生成   │  ← 一次输出多份 PRD
│   PRD        │
└──────┬──────┘
       ▼
┌─────────────┐
│ Agent：交叉    │  ← 多 Agent 从不同角色 Review
│  Review      │
└──────┬──────┘
       ▼
┌──────┴──────┐
│ 人：快速审查   │  ← 结构化 Checklist，聚焦关键决策
└──────┬──────┘
       ▼
┌──────┴──────┐          ┌─────────────┐
│ 开发 Agent   │ ◄──────► │ 人：持续审查  │
│  执行开发     │  同步修正  │  发现问题即改  │
└──────┬──────┘          └─────────────┘
       ▼
┌─────────────┐
│  CI/CD 自动   │  ← CI Spec 是质量护栏
│  验证与部署    │
└─────────────┘
```

### CI Spec 的核心地位

在极速迭代中，CI Spec 承担了过去"评审会议"的角色：

- **接口契约：** 定义模块间的输入输出规范，确保并行开发不冲突
- **质量基线：** 自动化测试覆盖核心路径，每次变更都有基本保障
- **回归防护：** 防止快速迭代中引入回归 Bug
- **Eval 集成：** 将 AI 能力的评测指标纳入 CI 流程，模型效果下降自动告警

## 总结

AI Native 产品工作流的本质不是"容忍混乱"，而是**用更快的反馈循环来替代更精确的前期规划**。不确定性不会消失，但可以通过以下方式管理：

| 不确定性 | 核心应对 |
|---------|---------|
| 模型输出不确定 | 封装层 + Guardrails + 多次采样 |
| 可实现性不确定 | Spike 验证 + Eval 驱动 + 降级方案 |
| PRD 内容不确定 | 结构化审查 + Agent 交叉 Review + 溯源标注 |

产品经理的角色从"精确的需求定义者"转变为"框架的设计者 + 质量的把关者 + 迭代的驱动者"。
