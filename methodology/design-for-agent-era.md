# Agent 时代的产品设计：当用户变成 Agent

当 agent 成为产品的直接体验者，产品设计从"打动人"变成"被 agent 选中 + 打动人"的双层博弈。

---

## 信任转移：产品最大的挑战

**以前**：用户体验产品 → 用户信任产品 → 用户付费
**以后**：agent 体验产品 → agent 向用户汇报 → 用户信任 agent → agent 代为付费

用户和产品之间多了一层 agent。功劳归 agent，不归产品。

---

## 产品的两层设计

产品经理需要同时设计两个界面：**agent 读的界面**和**人看的界面**。

| | 面向 Agent | 面向人 |
|------|-----------|--------|
| **目标** | 降低评估成本，让 agent 选你 | 制造感受，让人记住你 |
| **核心手段** | 结构化描述、标准化指标、可验证承诺 | 对比、缺口、归属、节奏 |
| **信任建立** | API 可靠性、数据一致性、协议兼容 | 视觉专业感、社会证明、品牌情感 |
| **转化驱动** | "这个方案最优" | "我想要" |
| **品牌感知** | Agent 记住能力标签 | 用户记住品牌 |

---

## 第一层：面向 Agent 的设计

Agent 跟产品打交道，不是"逛商场"，而是"批发采购"。

**Agent 的决策路径**：收到用户意图 → 检索候选方案 → 结构化对比 → 选最优 → 调用

### 一、让 agent 找到你

人靠搜索引擎和口碑发现产品，agent 靠**工具生态**（MCP、API marketplace、工具描述协议）。

工具描述不是营销文案，是机器可解析的能力声明：

```
# 人看的
"智能数据洞察平台，让数据驱动决策更简单"

# agent 读的
{
  "capability": "structured_data_analysis",
  "input": "CSV/JSON, max 10M rows",
  "output": "statistical_summary + anomaly_detection + trend_forecast",
  "latency": "< 30s for 1M rows",
  "price": "$0.01 per 1K rows",
  "reliability": "99.9% uptime, results deterministic"
}
```

上面那句话能打动人，但 agent 完全无法用它做决策。下面那段 agent 一秒就能对比十个竞品。

**以前你做 SEO 让产品在搜索引擎排前面，以后你要做 AEO（Agent Experience Optimization）让产品在 agent 的决策排序里排前面。**

### 二、让 agent 理解你

Agent 最怕模糊。人可以容忍"大约"、"视情况而定"，agent 不行。

- **能力边界要明确**：不说"支持多种格式"，说"支持 CSV、JSON、Parquet"
- **定价要可计算**：不说"联系销售获取报价"——对 agent 来说这等于"此路不通"
- **失败模式要可预期**：返回结构化错误码 + 建议替代方案，不是模糊的错误页面

**把产品所有"人情味模糊地带"翻译成"机器可执行的确定性"。**

### 三、让 agent 选择你

Agent 做选择靠可量化的对比：

- 第三方可验证的基准测试结果（不是自己吹的）
- 同类产品对比表（你主动提供；如果你不提供，agent 会自己去查，结果不可控）
- 用量和成本计算器（API 形式，让 agent 能直接算出"用你的产品处理这个任务要花多少钱"）

### 四、让 agent 复购你

Agent 记住调用历史。输出稳定、响应快、没有意外 → 下次优先选你。这就是 agent 的"信任"——不是情绪，是**正向反馈积累**。

反过来，API 偶尔抽风、返回格式不一致、价格波动不可预测 → agent 迅速降权。

**Agent 没有忠诚度，只有历史表现评分。**

## 第二层：穿透 Agent，直接打动人

关键洞察：**agent 替用户做了选择，但用户仍然有"使用时刻"。**

类比：你让助理帮你订餐厅。助理选了餐厅（结构化对比），但吃饭的是你。餐厅打动你的不是菜单上的数据，而是坐下来之后的灯光、第一口菜的味道。

策略：**在 agent 工作流中，设计"必须由人亲自体验"的时刻。**

### 手段一：体验入口——让 agent 交出一个"人应该自己看"的东西

Agent 调用 API 获取数据然后转述给用户——产品被 agent 消解了。但如果 API 返回的不只是数据，还有一个交互式报告链接？

```json
{
  "summary": { ... },          // agent 用这个做决策
  "report_url": "https://...", // 人看这个产生感受
  "share_card": { ... }        // 人传播这个
}
```

Agent 会说："分析结果在这里，建议你自己看一下这份报告。" 用户点开，看到精心设计的、有情绪节奏的页面——**这是你直接接触用户的唯一机会。**

**给 agent 结构化数据用于决策，给人一个体验入口用于感受。**

### 手段二：品牌寄生——在 agent 的转述里植入品牌语言

Agent 向用户汇报时会引用来源。如果你的输出自带有记忆点的表达，agent 很可能保留它：

```
# 普通输出（会被 agent 消化重述）
"用户活跃度下降 12%"

# 品牌化输出（agent 转述时保留品牌语言）
"【增长警报】用户活跃度本周下降 12%，连续第三周下滑，已触发关注阈值"
```

用户听到的就不只是数据，而是一个有品牌特征的洞察。**品牌声音寄生在 agent 的转述里。**

### 手段三：缺口钩子——用 agent 的嘴制造好奇

把情绪工具箱里的"缺口"设计进 API 返回：

```json
{
  "summary": "诊断完成，发现 3 个关键问题",
  "teaser": "其中 1 个问题可能在下周造成显著影响",
  "detail_url": "https://..."
}
```

Agent 转述 teaser，用户听到"下周造成显著影响"就会想亲自看看。**用 agent 的嘴制造缺口，用你的产品页面填补它。**

### 手段四：不可替代时刻——创造 agent 无法代替的体验

有些东西 agent 可以描述但无法代替你体验：

- **视觉冲击**：agent 可以说"数据有显著增长"，但你亲眼看到那条陡峭上升曲线时的感受不一样
- **交互探索**：agent 给你结论，但自己动手筛选、钻取、发现 insight 的成就感，agent 给不了
- **社交场景**：agent 帮你生成报告，但你把精心设计的分享卡片发到群里引发讨论的体验，属于人

**Agent 负责"帮你做完"，产品负责"让你想亲自看看"。**

---

## 产品架构总览

```
产品的两层设计：

Agent 层（让 agent 选择你）
├── 结构化能力描述（可检索、可对比）
├── 确定性接口（无模糊、可计算、可验证）
├── 稳定输出（建立正向反馈历史）
└── 竞争力可量化（基准、价格、可靠性）

Human 层（让人记住你）
├── 体验入口（报告页、可视化、交互探索）
├── 品牌寄生（在 agent 转述中植入品牌语言）
├── 缺口钩子（让 agent 的摘要引发人的好奇）
└── 不可替代时刻（视觉、交互、社交传播）
```

---

## 一个反直觉的结论

当所有产品都在讨好 agent 的时候，人反而会更信任那些直接打动自己的产品。

Agent 把所有人都推到功能和价格最优的方案上——唯一能让用户偏离"最优解"的，就是情绪。**在 agent 时代，情绪设计不会消失，它会变成差异化壁垒。**
