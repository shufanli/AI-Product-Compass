# 产品定义的钻探术：从模糊愿景到精准切片

> 复盘一次真实的产品定义对话（2026.03.31），记录老板如何一层层逼问，把一个模糊的"AI 代码审查产品"钻探到可执行的精准切片。这个过程本身就是方法论。

## 背景

我们在做一个 AI 代码审查产品。之前内部跑通了一个 use case（龙虾学校项目）：通过编排层让 AI 先做计划、分 sprint、每个 sprint 做真实浏览器验收，最终实现了"一次性交付无 bug"的效果。竞品有 Corridor（只做安全）和 CodeRabbit（全面 PR review）。

问题是：**这个产品到底是什么？卖什么？给谁？**

---

## 第一层：宏大叙事 — "AI 代码全生命周期质量把控"

**我的初始定位：**
> "AI 代码的全生命周期的质量把控。从计划到 sprint 分段到逐层审查到最终 PR review。"

**老板的第一刀：太重了。**

核心问题：
- 你在 skill 里写的规则，本质上就是你在做"AI 判断任务大小"这个产品功能，这给自己引入了不确定性
- 用户来你这里是为了确定性，你反而给自己加了有概率失效的环节
- **你卖给用户的是确定性，你不能自己引入不确定性**

**教训 1：不要在产品里做"可能失效的智能判断"，把不确定性留给用户自己的工具，你只对确定的事情负责。**

---

## 第二层：往回收 — "那就是 Bug 扫描工具"

讨论进入第二轮，开始做减法：

> 说话人 3："我感觉我们要解决的问题就是一个 bug 扫描工具。"

**老板追问：什么 bug 扫描？和 Claude Code 自带的 /review 有什么区别？**

> "AI 质量把控，你和 Claude Code 有啥区别？"
> "调一个 /review 命令，他们为什么不够用？我这其实——你做了一个让大家写代码效率更高的产品，别人根本不知道你在干嘛。"

**教训 2：如果你的一句话定位和已有免费方案听起来一样，用户就没有理由为你付费。"Bug 扫描"太宽泛，和 Claude /review 在用户心智里无法区分。**

---

## 第三层：痛点拆解 — "到底解决什么问题？"

老板反复追问："你到底在解决什么问题？"

我说："AI 没有办法一次性完成无 bug 的版本，需要人工反复核验打回。"

**老板指出：这是你最终想看到的结果，不是你解决的问题。**

> "它是你的那个间接的一个成果，你的直接产出还是说告诉了 Claude Code 这个软件哪些地方是点不通的，哪些地方会报错。"

**教训 3：区分"解决的问题"和"最终效果"。你做的事 ≠ 用户看到的效果 ≠ 你解决的底层问题。必须层层穿透到你真正在做的那个动作。**

---

## 第四层：找到真正的动作 — "模拟真实用户的自动验收"

讨论终于聚焦到了我们**实际做了什么**：

> 说话人 2："你做的是真实用户验收，你做的是一个验收——自动验收工具，模拟真实用户的自动验收工具。"

与 CodeRabbit 的核心区别浮出水面：
- CodeRabbit = 代码级别的静态扫描，PR 级别的审查，**没有真实用户测试**
- 我们 = 真实浏览器点击验证，端到端功能测试，**能发现 CodeRabbit 发现不了的 bug**

**教训 4：产品定义的突破口往往藏在"你和别人到底哪里不一样"。不是你比别人好多少，而是你做了别人根本没做的事。**

---

## 第五层：价值切片精准化 — "一个拳头就够了"

我试图把"真实浏览器验收"作为一个附加 feature，同时保持"全面 bug 扫描"的定位。

**老板否决：**

> "如果你这一个不行的话，你再加两三个更不行。这一个已经是你的拳头了。如果你一个拳头不行，你再加两个拳头也一点用都没有。"

一个好的切入点满足三个条件：
1. **足够精准** — 没有"和什么和什么"的复合条件
2. **可以直接赚钱** — 以它为点直接去撬动付费
3. **可以扩大深度** — 以它为点往外扩展

**教训 5：MVP 的价值切片只能有一个拳头。不是"我们做 A，顺便还能 B"，而是"我们就做 A，A 做到极致"。多个卖点 = 没有卖点。**

---

## 第六层：往下再钻一层 — "什么类型的 bug？什么类型的软件？"

确认了"功能测试"方向后，老板继续下钻：

| 维度 | 收敛结果 |
|------|----------|
| 测试类型 | 功能测试（不是安全、不是性能） |
| 覆盖问题 | AI agent 功能测试覆盖面不够 + 人跟不上 AI 写码速度 |
| 软件类型 | PC Web UI（不是移动端、不是 TUI） |
| 差异化 | Review-first：全程参与积累上下文，不是事后静态扫描 |

**教训 6：每一个维度都要做选择。不选 = 什么都做 = 什么都做不好。**

---

## 最终产品定义

**一句话：** 面向 Web 应用的 Review-first 全链路功能测试工具。

**解决的问题：** AI agent 写完代码后做 review，功能测试覆盖面有缺陷 — 因为不管是人还是现有 AI review，都缺乏从计划到编码全程的上下文。

**核心差异：** 从编码前就参与，全程积累上下文，做真实浏览器点击验收，而不是 PR 时才介入的静态代码扫描。

**实现路径：** Review-first 全链路 — 编排层在 AI 编码前就生成验收计划，编码过程中持续参与，PR 时做端到端功能验证。

---

## 未来扩展路径（三个维度）

```
                        测试维度扩展
                    功能测试 → 兼容性测试 → 性能测试 → 安全测试
                         ↑
                         |
  覆盖面扩展 ←———— 当前切入点 ————→ 软件类型扩展
  更多工具/策略提升              Web UI → 移动端 APP
  功能覆盖率
```

## 市场与验证

| 决策项 | 结论 |
|--------|------|
| 目标市场 | 北美（付费意愿强 100 倍，但仍需验证） |
| 目标用户 | SMB 老板 / CTO（不是一线开发者，开发者没动力为此付费） |
| 验证方式 | 先 DM 验证需求真伪，再做产品 |
| DM 渠道 | X、GitHub Discussion（Claude Code 等项目）、LinkedIn 社群 |
| 验证标准 | 愿不愿意付出时间（填表/加群）或金钱（$20 定金）|

**教训 7：用户说"我需要"不代表真需求。真需求的检验标准是用户愿意为之付出成本（时间或金钱）。50 人 DM 下来只有 2-3 人愿意付费 → 可能不值得做。**

## 复盘：产品定义的钻探方法论

回顾整个过程，老板用的方法可以提炼为一套可复用的钻探框架：

### Step 1 — 你在解决什么问题？
不接受"让代码更好"这种空泛回答。必须具体到：**谁**在**什么场景**下遇到了**什么具体困难**。

### Step 2 — 你和已有方案有什么不同？
不是"我比他好"，而是"我做了他根本没做的事"。如果答不出来，说明你还没找到差异化。

### Step 3 — 把"效果"和"动作"分开
"一次性交付无 bug"是效果，不是你做的事。你做的事是"在编码全程参与并做真实浏览器验收"。对外讲效果，对内清楚自己的动作。

### Step 4 — 每个维度做选择
测试类型、软件类型、目标用户、市场 — 每一个维度都必须做取舍。不选就是全选，全选就是什么都做不好。

### Step 5 — 一个拳头
最终切片只留一个核心价值点。不是附加功能的罗列，是一个让用户秒懂的、别人没有的、可以直接撬动付费的点。

### Step 6 — 先验证再做
定义清楚了不代表需求是真的。用最低成本（DM + 表单）验证用户是否愿意为此付出成本，再决定是否投入开发。

---

## 钻探过程的核心追问清单

每次做产品定义时，用以下问题自查：

- [ ] 我能用一句非技术语言说清楚这个产品做什么吗？
- [ ] 和市面上的免费方案（如 Claude /review）相比，用户为什么要付费给我？
- [ ] 我说的是"解决的问题"还是"最终效果"？两者有没有混淆？
- [ ] 有没有在每个维度上做了选择？（测试类型 / 软件类型 / 用户类型 / 市场）
- [ ] 核心价值点是一个还是多个？如果超过一个，哪个是真正的拳头？
- [ ] 有没有人愿意为这个东西付出时间或金钱？验证了吗？
