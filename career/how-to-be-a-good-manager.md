# 如何做好一个 Manager

> 关于带项目、带团队的真实体会。这是一篇会持续添加内容的笔记。

---

## 向上汇报：永远不要省

### 没有大卡点也要汇报

很多人会觉得：项目在顺利推进，没什么需要老板介入的，那就不用汇报了。这是巨大的误区。

只要你没有明确告诉老板"今天做了什么、有什么风险"，无论老板有没有在群里、有没有跟你私下聊过，他都会默认这件事在他视线之外。哪怕他看到群里你们在疯狂讨论，在他眼里也只是"这群人很忙"，不会自动转化为"这件事进展到哪里、有没有风险"。

后面一旦项目爆雷，他会找你：为什么之前没说？这种感觉非常糟糕，因为他会把"没说"理解成"瞒报"，无论你本意如何。

所以核心项目应该每天有向上汇报。日会上要明确总结："今天做了什么，卡点是 1、2、3。"日会上不说出来，别人就当你没有。

### 临近 Deadline 出现卡点时，反而要更频繁汇报

这是反直觉但极其重要的一点。

假设你的 deadline 是明天下午三点，有个卡点昨天就存在，今天还在，明天上午还会在。这种时候你心里会很挣扎——你解决不了，也不敢去找老板，因为你觉得一直不解决就显得自己能力不行。

但事实正好相反：**这种时候越要汇报，因为汇报本身才是能力的体现**。

如果一件事一直卡住，往往说明它的解决方案超出了你的认知范围。Manager 不可能解决所有事，能力之一就是知道什么时候该把问题升级。

频繁让领导介入，他不会觉得你能力不行——他会觉得你重视这件事，并且他会真的体会到这件事有多难。这两件事都对你有利。

作为 PM，遇到解决不了的问题时，**不要像一个开发那样守在工位等着**。开发的工作方式是"问题解决了才能往下走"，PM 不是。PM 的工作方式是用非常规手段把事情往前推，其中最重要的非常规手段就是频繁地让领导介入。

最坏的情况是 deadline 真的过了，但他在过程中一直在场——体感上是"这件事我看着他们拼了命，没办法"，而不是"这件事这么难，为什么之前没人告诉我？" 后者的反应会差一个数量级。

## 临近 Deadline 时的集中攻关

如果 deadline 已经在眼前，问题还没解决，**不要继续按平时的节奏推**。要换一种作战方式：

把所有项目相关的人全部召集到一个会议室，线下集中攻关。不是远程同步、不是分工各做各的，而是全部坐在一起，把这个问题死磕掉。

为什么这个做法有效？因为常规模式下，问题会在多个人之间漂移、被切碎、被延期。线下集中攻关把所有人的注意力锁在同一个问题上，沟通成本归零，决策可以即时做。

这件事必须 manager 主动发起。等所有人自己意识到问题严重性、自己组织起来，往往已经晚了。

## Manager 的杠杆在哪里：分配和验收

要学会做杠杆最高的事，**不要去做那些细小的具体任务**。

杠杆最高的事是两件：分配任务、做验收。

这两件事看起来像"没做什么"，但它们是把控全局的事，是只有 manager 能做的事。如果你扎进具体任务里面，你就退化成了一个普通执行者，整个项目就失去了管理者的视角。

### 不知道做什么的时候，先做这三件事

很多 manager 在拆解任务卡住、不知道下一步该做什么的时候，会随便找个具体任务做。这是错误的。

正确的做法是依次做这三件事：

**第一，确认所有人在做什么。** 不是听他们口头说"在做"，是真的去 check 他们手上是不是真的在推进你分配的任务，并且有阶段性结果。最怕的情况是下面有两个人私下对了一些问题、卡了很久，但你不知道——这种"超出你认知的事情"会直接撕裂项目进度。

**第二，梳理当前的卡点。** 如果有大卡点解决不了，参照前面"向上汇报"的原则，及时反馈。

**第三，做整体验收。** 如果项目在顺利推进，你应该去做整体验收，而不是扎进某个具体问题里。把整个事情的前后流程从头到尾走一遍——把自己当成一个用户，去体验最终的产物。这件事只有 manager 视角才能做好。

## 向下传递思维链路：不只是派活，还要传递背景

Manager 在信息流上是一个枢纽。老板的很多想法、产品的很多背景、决策背后的思考逻辑，往往只传达给了你一个人。这些信息如果不主动往下传，团队就只能在"知道做什么、不知道为什么做"的状态下工作。

这种状态会出问题。一个不知道"为什么"的执行者，遇到判断题的时候只能靠猜，做出来的东西往往偏离原本的意图。哪怕他本身能力很强，也用不到点子上。

所以作为 manager，及时把上面传达给你的思想往下传，是你的职责。**不只是结果，不只是派活——是产品的背景、老板的思维链路、为什么是现在做这件事、为什么是这个方案、为什么这件事重要。**

这种传达也不一定要通过开会。一起散步的时候、吃饭的时候、闲聊的时候，都是机会。各种各样的场景下，尽可能把项目背后的思维链路传播出去。这样团队成员对项目的理解才会和你保持一致，他们做出来的东西才会真的是你想要的。

不要假设"大家都懂"。绝大多数时候大家不懂——他们只是不好意思问。

## 跨部门协作：先经过对方的 Leader

作为 PM 需要 assign 任务给其他部门（比如开发部门）的具体成员时，有一条铁律：**先让对方的 Leader 知道，不要直接去 assign 那个具体的人。**

很多 PM 觉得"反正我知道这个开发能做这件事，直接给他就好了，省了一道流程"。这个思路会出问题。

最常见的问题是：你以为简单的事情，那个具体的开发可能觉得很麻烦；或者反过来，他觉得很麻烦的事，对他的 Leader 来说可能是个小事。**Leader 拥有的上下文比单个成员多得多**——他知道这个事情在他们整个团队里的优先级、知道是不是有更合适的人去做、知道当前的资源情况能不能扛得住。

直接绕过 Leader 派活，得到的反馈和判断一定是局部的、不准确的，而且 Leader 后面知道了还会觉得被绕过、被冒犯。

正确的做法是先和 Leader 沟通需求和时间预期，让他来决定具体派给谁、怎么排期。这样既保证了信息准确，也保护了协作关系。
