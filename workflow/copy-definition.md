# 文案定义工作流

一句话摘要：基于 A/B 测试实证数据，为产品各场景撰写高转化、低认知负担的 UX 文案。

> 来源：桌面文件 `ux-copy-patterns-reference.md`（GoodUI 141 个 UI 模式 + 638+ A/B 测试数据）+ `methodology/ux-review-spec.md` 文案审视部分 + `methodology/dont-make-me-think-notes.md` 第三定律

---

## 一、文案通用原则

### 5 条 UX 文案铁律

| # | 原则 | 说明 |
|---|------|------|
| 1 | **清晰 > 聪明** | 用户扫描不阅读，文案要一目了然 |
| 2 | **收益 > 功能** | "每周省 10 小时" > "自动化工作流引擎" |
| 3 | **具体 > 模糊** | "12,400 个团队在用" > "数千用户信赖" |
| 4 | **短 > 长** | 8 个词 = 100% 理解率；14 个词 = 90% |
| 5 | **主动 > 被动** | "我们找不到这个页面" > "页面未找到" |

### 文案精简三步法（Don't Make Me Think 第三定律）

1. 写完初稿
2. **删掉一半词**，检查信息是否完整
3. **再删掉剩下的一半**

**必须删除的内容：**
- "欢迎来到……，我们很高兴为您提供……"式的开场白（Happy Talk）
- "我们致力于为用户创造……"式的使命宣言
- 任何读的时候脑子里会产生"blah blah blah"感觉的文字
- 大段操作说明（用户不会在遇到问题之前读说明）

## 二、按场景撰写文案

### 2.1 CTA 按钮文案

**关键数据：**
- 第一人称代词：CTR 提升最高 90%
- 个性化 CTA：效果提升 202%（HubSpot）
- 具体 CTA vs 通用 CTA：转化提升 161%
- 时间紧迫语言：CTR 提升 35%

**模式 Checklist：**

- [ ] **用第一人称**："开始我的免费试用" 而非 "开始你的免费试用"
- [ ] **写收益不写动作**："获取 30% 折扣" 而非 "提交"
- [ ] **用动作动词**："加入等候名单" 而非 "提交"；"解锁完整权限" 而非 "升级"
- [ ] **适当加紧迫感**："立即获取" / "今天免费开始"

**反面示例 → 正面示例：**

| ❌ 不要 | ✅ 改为 |
|---------|---------|
| Submit | 加入等候名单 |
| Click here | 下载指南 |
| Learn more | 查看定价 |
| Register | 预订我的席位 |

---

### 2.2 登录/注册页文案

- [ ] **降低焦虑**：在关键字段旁加安抚文案
  - 邮箱字段下方："我们绝不会分享你的邮箱"
  - 付费字段旁边："暂时不会收费"
  - 名称字段旁边："之后可以随时修改"
- [ ] **协作语气 > 命令语气**："一起开始吧" > "创建账号"
- [ ] **先展示价值再索要信息**
- [ ] **加社会证明**："已有 10,000+ 团队在使用"

---

### 2.3 错误提示文案

- [ ] **说清楚发生了什么 + 怎么修**："银行卡被拒绝了，试试其他付款方式。"
- [ ] **绝不责怪用户**："这个邮箱格式不太对" 而非 "你输入了无效邮箱"
- [ ] **控制在 8 个词以内**：8 词 = 100% 理解率
- [ ] **就近显示**：错误信息紧贴出错字段，不要弹全局提示

---

### 2.4 空状态文案

三要素：**标题 + 说明 + 行动按钮**

- [ ] 标题告诉用户"这里是什么"
- [ ] 说明告诉用户"能做什么"
- [ ] 按钮引导用户"现在就做"

**示例：**
- Linear："还没有 Issue" / "追踪 Bug、任务和功能" / [新建 Issue]
- Todoist：任务清空时 → "享受剩下的一天吧"

---

### 2.5 Onboarding 文案

- [ ] **欢迎信息 = 问候 + 单个 CTA**，不要一次性塞太多
- [ ] **渐进式披露**：一次只展示一层信息
- [ ] **引导用户做第一个动作**，而不是给他看一堆介绍
- [ ] Tooltip 文案：标题 + 一句话描述，仅在用户触达该区域时出现

---

### 2.6 定价页文案

- [ ] **最多 3 个定价层级**：超过 4 个会导致决策疲劳，转化下降约 13%
- [ ] **标注推荐方案**："最受欢迎" / "推荐给大多数团队"
- [ ] **锚定定价**：先展示高价，让目标价位显得划算（GoodUI 测试：+48% 购买）
- [ ] **一次性付款措辞**："一次付清 ¥299" 而非 "全额支付"（GoodUI 模式 #132）

---

### 2.7 社会证明文案

- [ ] **用具体数字**："12,400 个团队在用" 而非 "数千用户信赖"
- [ ] **用收益型推荐语**："帮我把设计时间缩短了 30%" 而非 "这个产品很好用"
- [ ] **信任标识**：安全认证、G2 评分、知名客户 Logo

---

### 2.8 弹窗/确认对话框文案

- [ ] **用动词标签，不用"是/否"**："删除项目" / "保留项目" 而非 "确定" / "取消"
- [ ] **说清后果**："删除 #频道名？此操作无法撤销。所有消息将被永久删除。"
- [ ] **按钮文案匹配触发动作**：点了"删除账号" → 弹窗按钮写"删除我的账号"，不写"确认"
- [ ] **破坏性操作加摩擦**：输入项目名称确认删除

---

### 2.9 成功/确认消息

- [ ] **确认 + 引导下一步**："订单已确认。在这里追踪物流。"
- [ ] **避免"成功"二字**："付款已收到" 而非 "付款成功！"
- [ ] **控制标题在 3-4 个词**

---

### 2.10 404 页面

- [ ] **承认 + 引导 + 品牌调性**
- [ ] **提供明确的导航出路**：返回首页 + 搜索框

---

### 2.11 加载状态

- [ ] **短等待（<10s）**：用骨架屏，不用转圈
- [ ] **长等待**：说明正在做什么（"正在设置你的工作区…"）
- [ ] **超长等待**：渐进式文案（"准备中…" → "快好了…" → "还需要一会儿…"）

## 产出物

完成文案工作后，应输出：

1. **各场景文案稿**——按上述分类逐一撰写
2. **文案精简确认**——每条文案是否经过"删一半再删一半"检验
3. **A/B 测试计划**——关键文案是否安排了数据验证
