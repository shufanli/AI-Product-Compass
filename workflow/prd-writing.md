# PRD 撰写工作流

一句话摘要：AI Native 时代的 PRD 不追求一次到位，而是框架先行、AI 生成、多角色交叉审查、边跑边改。

> 来源：`methodology/ai-native-product-workflow.md`

---

## 核心理念

### 范式转变

| 维度 | 传统 PRD | AI Native PRD |
|------|---------|--------------|
| 撰写方式 | 人精确定义每一个交互细节 | 人定框架，AI 批量生成，Agent 交叉 Review |
| 启动时机 | 等 PRD 完全 ready 才启动开发 | 不等完全 ready 就启动，边跑边改 |
| 修改成本 | 高（人工撰写、评审会议） | 极低（AI 生成和修改速度远超人工） |
| 流程模式 | 线性：需求确认→设计→开发→测试→上线 | 并行：PRD 生成 / 开发 / Review 同时推进 |

### 修改成本极低 = 不必追求一次到位

AI 生成和修改 PRD 的速度远超人工，核心变化是：
- PRD 完善和开发可以**同步进行**
- 发现问题即时修正、即时同步
- CI Spec 成为质量护栏，替代传统评审会议

---

## Step 1：人定框架

在让 AI 生成 PRD 之前，人必须先定义清楚以下内容：

- [ ] **产品愿景**：这个产品/功能要解决什么问题？（来自产品定义 skill 的输出）
- [ ] **核心功能**：必须有的功能是什么？（不超过 3 个）
- [ ] **约束条件**：技术限制、时间限制、资源限制是什么？
- [ ] **验收标准**：怎么判断这个功能做好了？

---

## Step 2：AI 批量生成 PRD

将框架交给 AI，一次输出多份 PRD 方案。

**给 AI 的输入应包含：**
- Step 1 中定义的框架
- 目标用户画像
- 竞品分析结论（来自竞品分析 skill 的输出）
- 已知的技术约束

---

## Step 3：多角色交叉审查

### 三个必答审查问题

对每份 PRD，强制回答以下三个问题：

| # | 问题 | 审查焦点 |
|---|------|---------|
| 1 | **对不对** | 现有逻辑是否自洽、是否有幻觉 |
| 2 | **该不该** | 这个需求的优先级和 ROI 是否合理 |
| 3 | **全不全** | 是否遗漏了边界场景、异常处理、关联需求 |

### 多角色 Review

让多个视角审查同一份 PRD，弥补单一视角的盲区：

| 角色 | 审查重点 |
|------|---------|
| **用户视角** | 这个功能用户真的需要吗？使用流程自然吗？ |
| **开发视角** | 技术上可实现吗？有没有被忽视的技术约束？ |
| **测试视角** | 边界场景是否覆盖？异常处理是否定义？ |
| **业务视角** | ROI 是否合理？与业务目标是否对齐？ |

### 溯源标注

- [ ] PRD 中的关键结论是否标注了依据来源？
- [ ] 无来源的结论是否被重点审查？

---

## Step 4：处理三大不确定性

AI 产品的 PRD 必须额外处理三类不确定性：

### 4.1 大模型输出不确定

| 应对策略 | 说明 |
|---------|------|
| 输出封装层 | 在模型原始输出和用户之间建立封装层（格式化、校验、兜底逻辑） |
| Guardrails | Output Parser、内容过滤、格式约束，收窄输出空间 |
| 多次采样 + 筛选 | 关键输出多次生成、自动评估、择优展示 |
| 用户预期管理 | 产品设计层面让用户理解 AI 输出的概率性（"重新生成"按钮、置信度标识） |

**PRD 中必须写清：**
- [ ] 模型输出的封装层逻辑是什么？
- [ ] Fallback（兜底）方案是什么？
- [ ] 用户如何感知到 AI 的不确定性？

### 4.2 可实现性不确定

| 应对策略 | 说明 |
|---------|------|
| Spike 验证前置 | 对不确定的 AI 能力，先用最小 Prototype 验证可行性，再写完整 PRD |
| 分级承诺 | 将功能拆为"确定可实现"和"需要验证"两类 |
| Eval 驱动开发 | 建立自动化评测体系（准确率、延迟、成本） |
| 降级方案前置 | 每个 AI 能力点都预设降级策略 |

**PRD 中必须写清：**
- [ ] 哪些功能点需要 Spike 验证？验收标准（Eval Metrics）是什么？
- [ ] 每个 AI 能力的降级方案是什么？

### 4.3 审查者盲区

AI 生成的 PRD 可能包含幻觉，人作为审查者（而非撰写者）容易忽略"该不该做"和"还能做什么"。

**应对：**
- [ ] 是否完成了 Step 3 的三个必答问题（对不对、该不该、全不全）？
- [ ] 是否经过多角色交叉审查？
- [ ] 无来源的结论是否被标注并重点审查？

---

## Step 5：CI Spec 定义

在极速迭代中，CI Spec 承担了传统"评审会议"的角色：

- [ ] **接口契约**：模块间的输入输出规范是否定义清楚？
- [ ] **质量基线**：自动化测试是否覆盖核心路径？
- [ ] **回归防护**：是否有机制防止快速迭代引入回归 Bug？
- [ ] **Eval 集成**：AI 能力的评测指标是否纳入 CI 流程？

---

## Step 6：同步开发，持续修正

PRD 交付后进入并行模式：

```
┌──────────┐          ┌─────────────┐
│ 开发执行   │ ◄──────► │ 人：持续审查  │
│           │  同步修正  │  发现问题即改  │
└──────┬───┘          └─────────────┘
       ▼
┌─────────────┐
│  CI/CD 自动   │  ← CI Spec 是质量护栏
│  验证与部署    │
└─────────────┘
```

**关键原则：**
- 发现问题即时修正、即时同步给开发
- 不要攒一堆问题再集中改——那是传统模式
- CI 自动验证是底线保障

---

## PRD 输出结构

最终 PRD 应包含以下部分：

| 部分 | 内容 |
|------|------|
| 背景与目标 | 解决什么问题、为谁解决、成功指标 |
| 核心功能 | 功能列表 + 优先级（P0/P1/P2） |
| 用户流程 | 主流程 + 异常流程 + 边界场景 |
| AI 能力说明 | 模型选择、Prompt 策略、封装层、Guardrails、降级方案 |
| Eval 标准 | 每个 AI 能力的量化验收指标 |
| CI Spec | 接口契约、自动化测试范围、Eval 集成方案 |
| 数据埋点 | 需要哪些埋点（对接上线前埋点 skill） |
| 里程碑 | Spike 验证节点、MVP 交付节点、正式上线节点 |
