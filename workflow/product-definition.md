# 产品定义工作流

一句话摘要：从真实问题出发，一句话定义清楚，小而精准地打向市场，并通过结构性风险自检确保不在死亡线上。

> 来源：`methodology/how-to-think-of-a-feature.md` + `case-studies/ppt-disappeared-apps-ai-opportunities.md`（产品生死线自检）

---

## 工作流步骤

### Step 1：找到背后的真问题

所有 Feature 必须从真实世界的问题出发，不能为了形式、为了产品而做产品。

**自检三问：**

- [ ] 这个问题在真实用户的生活/工作中是否真实存在？
- [ ] 用户现在是如何解决这个问题的（即使用很笨的方式）？
- [ ] 如果没有你的产品，用户会感到痛苦还是只是"还好"？

**警惕表面问题：** 表面上是"提供情绪价值"，但情绪价值背后也有更本质的真问题。参考奢侈品的底层逻辑——"不用开口，别人就知道我的阶级"，这才是真问题。

---

### Step 2：用一句话定义清楚问题

真问题一定可以被一句话定义清楚。

**警惕"糅合陷阱"：** 不要把几个单独不 work 的想法强行糅合在一起。一个问题本身不 work，糅合之后依然不 work，甚至更复杂。

**一句话定义之后，同步评估三个维度：**

| 维度 | 问题 |
|------|------|
| 病毒传播 | 用户会自发分享给别人吗？ |
| 商业化 | 有没有清晰的变现路径？ |
| 上线速度 | 能快速验证、快速上线吗？ |

---

### Step 3：确保足够小、足够精准

Feature 不需要很大，一个很小的点，只要足够精准，就能打向市场。

**小而精的优势：**
- 降低开发成本，缩短验证周期
- 用户的理解成本低，更容易传播
- 聚焦一个场景，体验更极致

**反例：** 大而全的功能堆砌，往往导致每个功能都平庸，没有一个能让用户记住。

---

### Step 4：产品生死线自检

在定义完产品后，用五种结构性死法框架检验产品的生存能力。任何一个答案亮红灯，都需要重新审视定位。

> 来源：从移动互联网消失的 App 中提炼的结构性风险框架。猎豹（6 亿月活→市值跌 90%）、豌豆荚（10 亿估值→2 亿卖身）、WiFi 万能钥匙（9 亿用户→被遗忘）、美图（市值千亿→跌 97%）、墨迹天气（6.5 亿用户→IPO 被否）——全部不是因为"产品做得不好"，而是结构性力量变了。

**五问自检：**

- [ ] **系统吞噬风险**：下个操作系统更新会自带你的功能吗？
  - 亮红灯 → 你做的是通用能力。出路：做系统不会做的**垂直深度**

- [ ] **巨头碾压风险**：模型厂商/平台方明天做了你的功能，用户会走吗？
  - 亮红灯 → 你是 wrapper。出路：拥有**场景和数据**，不是模型调用能力

- [ ] **需求消失风险**：AI 免费那天，你的产品还值钱吗？
  - 亮红灯 → 你在卖"AI 能力"本身。出路：卖 **AI + 场景理解 = 结果**

- [ ] **超级 App 虹吸风险**：用户一句 prompt 能替代你吗？
  - 亮红灯 → 你是单功能工具。出路：做一句 prompt 解决不了的**重交付**

- [ ] **工具宿命风险**：用户用第 100 次和第 1 次，体验有区别吗？
  - 亮红灯 → 用完即走。出路：**记忆和个性化**——越用越懂你

**存活的产品共性：** 不是在卖 AI 能力，而是在卖 AI + 场景 = 不可替代的结果。（Cursor、Midjourney、Perplexity、Notion AI 皆如此。详见竞品分析 skill。）

---

## 产出物

完成以上步骤后，应输出：

1. **一句话问题定义**——用户的真问题是什么
2. **一句话产品定义**——我们用什么方式解决它
3. **三维评估**——病毒传播 / 商业化 / 上线速度各自的判断
4. **最小功能边界**——这个 Feature 的最小可交付范围是什么
5. **生死线自检结果**——五问中哪些亮红灯、对应的出路是什么
