# 上线前埋点和数据漏斗准备工作流

一句话摘要：产品上线不是终点，而是数据验证的起点——发布前埋好每一层漏斗，发布后才能快速定位问题。

> 来源：`methodology/launch-data-funnel.md`

---

## 核心理念

### 不要只看绝对数，要看漏斗转化率和增长斜率

- **转化漏斗是否足够宽** — 每一层的转化率是否健康
- **增长斜率是否够大** — 用户是否在自发裂变（100 人能否一天变 1,000 人）

### 触达 ≠ 使用，要逐层折损估算

向 2,000 人发消息 ≠ 2,000 人看到。必须逐层折损：

| 折损层级 | 折损因素 | 估算系数 |
|----------|----------|----------|
| 第 1 层 | 目标用户占比（群内精准用户比例） | ~30% |
| 第 2 层 | 活跃留存（当前仍在使用/关注） | ~10% |
| 第 3 层 | 消息阅读率（时间段、平台） | ~25%（飞书全员消息） |
| 第 4 层 | 理解并行动（文案是否清晰） | 再打折 |

**示例**：500 人群 → 150 精准用户 → 15 人留存活跃 → ~10 人看到消息 → 最终进入产品的个位数是正常的。

---

## Step 1：定义完整转化漏斗

发布前必须画出一条**完整的主转化链路**，确保每一层都有埋点：

```
曝光/触达
  → 落地页访问（UV/PV）
    → 核心动作完成（如：完成答题）
      → 结果页到达
        → 转化动作（升级/付费/分享）
          → 转化成功
```

---

## Step 2：埋点 Checklist

### 主转化链路

| 漏斗层级 | 必须埋的点 | 常见遗漏 |
|----------|-----------|---------|
| **触达层** | 消息发送数、曝光数、阅读率 | 只看发送数不看阅读率 |
| **落地页** | UV、PV、来源渠道（UTM） | 不区分渠道来源 |
| **核心体验** | 开始操作数、完成操作数 | 只埋完成不埋开始，无法算完成率 |
| **结果/价值页** | 页面到达数、停留时长 | 缺失此层导致漏斗断裂 |
| **转化动作** | 按钮点击数、表单提交数 | 只埋最终成功不埋中间步骤 |
| **转化成功** | 付费成功数、分享成功数 | 分享点击 vs 分享成功未区分 |
| **裂变回流** | 被分享者的注册/激活数 | 完全缺失，无法衡量裂变效果 |

### 分支链路

- [ ] **分享链路**：分享按钮点击 → 分享成功 → 被分享者访问 → 被分享者注册/激活
- [ ] **付费链路**：价格页浏览 → 付费按钮点击 → 填写信息 → 支付成功
- [ ] **多入口区分**：同一动作的多个触发入口（如页面右上角 vs 弹窗内）需区分来源

## Step 3：埋点质量验证

- [ ] 所有埋点已上线并经过 **QA 验证**（非只看代码提交）
- [ ] 埋点数据**交叉校验**通过（如：分享成功数 ≤ 分享点击数）
- [ ] 内部测试用户已打标，数据可过滤

---

## Step 4：数据看板搭建

- [ ] 看板已搭建完成
- [ ] 看板链接已确认可正常访问（不同角色均可打开）
- [ ] 看板数据正在正常更新（非静态 / 非挂掉状态）
- [ ] 已设置关键指标的基准值 / 预期值

---

## Step 5：触达估算

- [ ] 发布渠道已确认（社群 / 消息推送 / 邮件 / 其他）
- [ ] 目标用户画像清晰，已估算精准用户占比
- [ ] 已做触达折损估算：

| 折损层级 | 预估值 |
|----------|--------|
| 目标用户占比 | ___% |
| 活跃留存率 | ___% |
| 消息阅读率 | ___% |
| 理解并行动率 | ___% |
| **预估实际进入产品人数** | **___ 人** |

---

## Step 6：发布文案检查

- [ ] 文案说清三件事：**做了什么** / **对用户有什么用** / **怎么参与**
- [ ] 文案已经内部人员试读确认「看得懂」
- [ ] 文案中包含明确的行动引导（CTA）

---

## Step 7：技术 & 稳定性

- [ ] 功能已在预发布环境验证通过
- [ ] 已做基本压测 / 评估并发承载能力
- [ ] 关键页面加载正常（无卡死 / 丢失元素 / 缓存问题）
- [ ] 错误告警 / 监控已配置

---

## Step 8：发布后对齐安排

- [ ] T+1 数据对齐会已排期（时间：___）
- [ ] 参与人确认：产品 / 研发 / 增长
- [ ] 已明确对齐会重点关注项

## Go / No-Go 决策

| 检查项 | 状态 |
|--------|------|
| 埋点完备 | ⬜ Go / ⬜ No-Go |
| 看板可用 | ⬜ Go / ⬜ No-Go |
| 触达估算合理 | ⬜ Go / ⬜ No-Go |
| 文案清晰 | ⬜ Go / ⬜ No-Go |
| 技术稳定 | ⬜ Go / ⬜ No-Go |
| **最终决定** | **⬜ Go / ⬜ No-Go** |

## 常见踩坑

| 踩坑场景 | 问题 | 正确做法 |
|----------|------|---------|
| 发布后才补埋点 | 丢失早期最宝贵的数据 | 埋点与功能同步开发、同步上线 |
| 看板上线没验证 | 发布后发现看板挂了 | 发布前确认看板可用 |
| 分享"点击"和"成功"不区分 | 数据矛盾，无法定位 | 分别埋点，明确口径 |
| 只看「发了多少人」 | 高估触达，误判产品问题 | 逐层折损估算实际触达 |
| 忽视消息阅读率 | 把渠道问题误判为产品问题 | 拿到阅读率数据，区分渠道和产品 |
| 不过滤内部测试数据 | 早期数据被污染 | 内部用户打标，数据可过滤 |
| 发布文案不清晰 | 误判为无需求 | 文案说清：做了什么、有什么用、怎么参与 |
